# Create Template (L2)

## Two template families

| Family         | Dir                            | Deployed by         | When to use                                      |
| -------------- | ------------------------------ | ------------------- | ------------------------------------------------ |
| **bun-create** | `templates/bun-create/<name>/` | `bun create <name>` | Bootstrap a project from anywhere (local or npm) |
| **scaffold**   | `templates/scaffold/`          | `kimi-fix <path>`   | Inject files into new/existing projects          |

## bun-create templates

### Structure

```
templates/bun-create/<name>/
└── package.json    ← minimal: bun-create.postinstall hook
```

The template must be **maximally minimal**. Only `package.json` is required. Everything else is generated by `kimi-fix` via the `bun-create.postinstall` hook.

### package.json format

```json
{
  "name": "<name>",
  "bun-create": {
    "postinstall": ["bun install -g github:brendadeeznuts1111/kimi-toolchain", "kimi-fix ."]
  }
}
```

**Rules:**

- **Zero dependencies.** If `package.json` has `dependencies` or `devDependencies`, `bun create` runs the **auto-detected npm client** (pnpm/yarn/npm) for install — NOT bun — and the hardened `bunfig.toml` won't exist yet. Keep it empty.
- **No README.md.** `kimi-fix` generates the project README from `templates/scaffold/README.md`. If you include a README in the bun-create template, `kimi-fix`'s `!pathExists()` guard skips it.
- **Postinstall is idempotent.** `bun install -g` is a fast no-op if already installed.

### Registration

1. Create the directory: `templates/bun-create/<name>/`
2. Add `package.json` with `bun-create.postinstall`
3. Document in `TEMPLATES.md` § bun create Template
4. **Optional:** publish as `create-<name>` to npm for zero-install `bun create <name>`

### Activation

```bash
# Local dev (from repo root):
bun create kimi-toolchain <dest>

# Global (available everywhere):
cp -r templates/bun-create/<name> ~/.bun-create/<name>

# Or via BUN_CREATE_DIR:
export BUN_CREATE_DIR="$HOME/kimi-toolchain/templates/bun-create"
```

### Adding a new scaffold template

1. **Create the file** in `templates/scaffold/`. Keep it minimal — `kimi-fix` will copy it verbatim. Use `{{PROJECT_NAME}}` for project-name placeholders (rendered by `renderTemplate()`).

2. **Register in `scaffold-templates.ts`:**

```ts
// src/lib/scaffold-templates.ts
export const MY_TEMPLATE = load("my-template.json");
```

3. **Teach `kimi-fix` to write it** in `src/bin/kimi-fix.ts`:

```ts
const myPath = join(project, "my-template.json");
if (!pathExists(myPath)) {
  stepLog("my-template", "creating my-template.json...");
  await writeFile(myPath, MY_TEMPLATE, dryRun);
}
```

**Critical:** Always use `!pathExists()` guard — `kimi-fix` is non-destructive. Never overwrite existing files. If the file needs to be always-updated (not just created), add an explicit flag or content check.

4. **Add to scaffold-doctor checks** in `src/lib/scaffold-doctor.ts`:

```ts
{ name: "my-template", rel: "my-template.json" },
```

And update the test constants in `test/scaffold-doctor.unit.test.ts`:

```ts
const EXPECTED_FILE_NAMES = [
  // ... existing entries
  "my-template",
];
const FILE_PATHS: Record<string, string> = {
  // ... existing entries
  "my-template": "my-template.json",
};
```

5. **Update `TEMPLATES.md`** — add a section documenting the template, its purpose, and any placeholders.

6. **If it's a documentation template** (like `adr-template.md`), add it to `LOCAL_DOC_REFERENCES` in `src/lib/canonical-references.ts` and regenerate: `bun run references:generate`.

### Profile differentiation

| Profile           | Flag                                  | Templates used                    | Extra files                            |
| ----------------- | ------------------------------------- | --------------------------------- | -------------------------------------- |
| **app** (default) | `kimi-fix <path>`                     | All config + docs + source layers | None                                   |
| **toolchain**     | `kimi-fix <path> --profile toolchain` | All of above + scripts layer      | 6 scripts + `dx.config.toolchain.toml` |

Templates shared across both profiles just exist in `templates/scaffold/`. Toolchain-only files are gated in `kimi-fix.ts` with `if (profile === "toolchain")`.

## Collision rules

### bun init collision (the big one)

`bun init` creates: `tsconfig.json`, `index.ts`, `README.md`, `.gitignore`, `CLAUDE.md`, `.cursor/rules/*.mdc`.

`kimi-fix` also creates hardened versions of these files. If `bun init` runs first, `kimi-fix`'s `!pathExists()` guard skips them — leaving the basic versions.

**Solution:** The `bun create` template's postinstall does **not** call `bun init`. `kimi-fix` handles everything, including `@types/bun` installation. Only use `bun init` separately for its AI agent rule files (`CLAUDE.md`, `.cursor/rules/`).

### kimi-fix skip rule

`kimi-fix` writes a file **only if it doesn't exist**. This is intentional — it never overwrites user customizations. If you need a template that updates on every `kimi-fix` run, add an explicit overwrite flag or hash check.

### Governance collision

`kimi-governance fix` (called inside `kimi-fix`) generates its own `README.md` and `CONTRIBUTING.md`. To avoid collision, `kimi-fix` writes `README.md` **before** delegating to governance tools.

## Verification checklist

After creating/modifying a template:

```
1. bun run typecheck          ← TypeScript compiles
2. bun run format             ← oxfmt passes
3. bun test test/scaffold-*   ← scaffold tests pass
4. kimi-fix .tmp-test && find .tmp-test -type f | sort  ← files land correctly
5. rm -rf .tmp-test
6. kimi-doctor --quick        ← no health regressions
7. git add + commit
```
